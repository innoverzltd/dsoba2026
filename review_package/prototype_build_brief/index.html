Clickable vs Display-only Scope

Must be clickable

  • Public event guest registration from event detail to confirmation.
  • Optional public account creation/login path.
  • Portal dashboard to membership status and payment states.
  • Trial-to-Life payment happy path and inactive state demonstration.
  • Annual Dinner individual ticket/table selection to confirmation.
  • Member event registration to ticket/QR view.
  • Support case creation and case status.
  • Booking facility selection to request status.
  • Directory search to full member profile view.

Display-only or light interaction

  • About, legal, news article, chapter detail content patterns.
  • Public site visual design variants before final route selection.
  • Admin dashboard summary and report/audit preview.
  • Notification inbox and preference toggles.
  • CMS editing, real payment, real WhatsApp, and backend operations are not included.

Confirmed Decisions vs Client Review Items

Confirmed for Prototype

  • Public event registration does not require account creation.
  • Public account remains Phase 1 scope but is optional for public event registration.
  • Youth Members auto-convert to Trial Members at age 18.
  • Trial Members must become paid Life Members at age 28; no grace period; unpaid status becomes inactive.
  • WhatsApp is notification-only in Phase 1.
  • Annual Dinner supports both individual tickets and table bookings.
  • Seat/table assignment happens after registration closes.
  • Booking is Phase 1 but only appears inside the portal.
  • Orders/e-shop should not appear in this phase.
  • Admin launch roles are Admin and Staff, with admin type controlling page/action rights.
  • All admin actions require audit reason capture, including view/create/edit/delete.

Still Needs Client Review

  • Final public site visual direction: Classic, Modern Member Platform, Community & Events-led, or hybrid.
  • Exact public section content, including which chapters, committees, galleries, and legacy archives launch first.
  • Whether bilingual content is required in Phase 1 and how it affects navigation length.
  • Exact data fields visible in member directory, even though current decision says all fields visible.
  • Final privacy and consent wording for directory visibility and personal data usage.
  • Exact admin type matrix and which pages/actions each admin type can access.
  • Exact report pack needed at launch.
Privacy note: "All members and all fields visible" is currently treated as a prototype rule, but it should be separately reviewed against consent and personal data requirements before implementation.

Recommended Next Step After This Brief

Once this build brief is accepted, the next artifact should be a layout direction package: public site design system options first, then portal shell layout. Only after that should implementation of the clickable web prototype begin.
