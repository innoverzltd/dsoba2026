Interaction Principles

These rules should guide the clickable prototype so that AI or developers do not invent behavior that contradicts the sitemap or UR decisions.

Clickable Display-only Route State change Confirmed Client review Risk

Public site confirms structure

Public pages should show sitemap coverage, navigation, content modules, and entry points. Only event registration and membership application entry need deeper click-through.

Portal confirms behavior

Portal screens should show role/state-specific behavior: applicant, Youth, Trial, Life, public account, inactive, pending payment, confirmed event, waitlist, and booking states.

Admin remains thin

Admin preview only demonstrates what staff sees after user actions. It should include admin type rights and audit reason capture, but no full CRUD build in this phase.

Prototype Route Map

Recommended routes for the clickable prototype. Routes are not final production URLs, but they should be stable during prototype review.

Shared Flow Registry

This spec uses the same 12 swimlane flows as the Prototype Build Brief. Screen cards reference these flows; they do not create additional flows.

Public Site Screen Interaction Specs

Public site screens are mainly layout and content-pattern screens. Only screens that start a flow should contain deeper interactions.

Membership Portal Screen Interaction Specs

Portal screens should be built around persona/state switching so stakeholders can see how the same interface behaves for different account types.

Admin Preview Screen Interaction Specs

Admin screens are supporting evidence for the front-end flows. They show how staff sees queues, payments, permissions, and audit logs.

Confirmed Decisions And Review Items

Confirmed behavior to implement in prototype

  • Public event registration can be completed as guest without account creation.
  • Public account exists in Phase 1, but it is optional for public event registration.
  • Youth Member auto-converts to Trial Member at age 18.
  • Trial Member must pay at age 28; no grace period; unpaid becomes inactive.
  • WhatsApp is notification-only in Phase 1.
  • Annual Dinner supports individual ticket and table booking; seat/table assignment happens after registration closes.
  • Booking is Phase 1 but only visible inside the portal.
  • Orders/e-shop is not shown in this prototype phase.
  • Admin and Staff roles use admin type page/action rights. All admin actions require audit reason capture.

Still needs client review before build freeze

  • Final public design direction and whether it combines multiple visual routes.
  • Which chapters, committees, galleries, newsletters, and archives appear at launch.
  • Bilingual content requirement and navigation impact.
  • Exact member fields visible in Directory, despite current all-fields-visible decision.
  • Privacy and consent wording for Directory visibility and personal data usage.
  • Exact admin type permission matrix and launch report pack.
